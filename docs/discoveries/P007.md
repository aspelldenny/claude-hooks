## Test Strategy

**Routing unit tests** (7 new in `tests/mcp_handshake.rs`): use `tools/call why_blocked` via MCP `call_why_blocked()` helper. Test cases chosen to be deterministic (no fs marker / no network):

- `Edit` + `.env.local` → `hook="block_env_edit"`, `blocked=true` (pure path-regex, no fs)
- `Edit` + `.env.example` → `hook="block_env_edit"`, `blocked=false` (allowlist, no fs)
- `Write` + `.env.local` → `hook="block_env_edit"`, `blocked=true` (Write routes same as Edit)
- `MultiEdit` + `.env.local` → `hook="block_env_edit"`, `blocked=true`
- `NotebookEdit` + `.env.local` → `hook="block_env_edit"`, `blocked=true`
- `WebFetch` → `hook="none"`, `blocked=false`, reason contains `"no hook matches tool WebFetch"`
- `Bash` + `"echo hi"` → `hook="block_unsafe_merge"`, `blocked=false`

**Bash + `"echo hi"` rationale:** `parse_merge_pr("echo hi")` returns `None` (no `gh pr merge \d+` match) → `block_unsafe_merge_decide` returns `ALLOW` immediately at step 2, before any gh shell call. Deterministic — no network needed.

**`Read`/`Glob` route (architect_guard — fs-dependent):** `architect_guard_decide` reads `.sos-state/architect-active` marker from real fs. Result is environment-dependent. This route is already covered by the existing `decide_architect_guard_no_marker_allows_src` and `decide_architect_guard_marker_src_blocks` tests (P006) which use proper `CLAUDE_PROJECT_DIR` isolation via temp dir. Not duplicated here to avoid nondeterminism in `why_blocked` tests. Discovery note: if serve runs outside an architect-active context (marker absent), `Read`/`Glob` via `why_blocked` returns ALLOW — correct and honest behavior.

**Handshake 5-tool smoke:** `mcp_serve_tools_list_returns_5_tools` (renamed from `_4_tools`). Assert all 5 tool names: `architect_guard`, `block_env_edit`, `block_unsafe_merge`, `session_banner`, `why_blocked`.
