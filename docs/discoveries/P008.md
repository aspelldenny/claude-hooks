## Việc 1 — get_info refactor (`server_handler` flag → explicit impl)

**Approach used:** `#[tool_router]` (no flag) on `impl HooksServer` + separate `#[tool_handler] impl ServerHandler for HooksServer { fn get_info() }`.

**Exact import paths (verified by compiler):**
- `ServerHandler`: `rmcp::ServerHandler` (re-exported at crate root from `rmcp::handler::server`)
- `tool_handler`: `rmcp::tool_handler` (re-exported via `rmcp_macros::*` at crate root)
- `ServerInfo`: `rmcp::model::ServerInfo` (type alias `InitializeResult` at `model.rs:893`)
- `Implementation`: `rmcp::model::Implementation` (`struct` at `model.rs:994`)

**`get_info()` impl:** Used `ServerInfo::default().with_server_info(Implementation::new("claude-hooks", env!("CARGO_PKG_VERSION")))` — builder pattern matches rmcp 1.7 test examples (`test_tool_macros.rs:559`). Simpler and more correct than `..Default::default()` struct literal (avoids `#[non_exhaustive]` issue on `Implementation`).

**`#[tool_handler]` wiring:** The macro reads the struct field named `tool_router: ToolRouter<Self>` and generates `list_tools` / `call_tool` implementations on `ServerHandler`. No arg needed when field is named `tool_router` (default field name). If field were named differently, `#[tool_handler(router = self.other_field)]` form would be required (confirmed in `test_tool_macros.rs:28`).

**Why `server_handler` flag produced wrong name:** `#[tool_router(server_handler)]` generated a `ServerHandler` impl that called `Implementation::from_build_env()` — but `from_build_env()` uses `env!("CARGO_CRATE_NAME")` which expands at the rmcp crate's build site (crate name = "rmcp"), not at the user crate's build site.

**Real handshake output (after fix):**
```
{"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2024-11-05","capabilities":{},"serverInfo":{"name":"claude-hooks","version":"0.1.0"}}}
```
(version shows 0.1.0 because build was before Cargo.toml bump; after bump will be 0.8.0 — `env!("CARGO_PKG_VERSION")` is compile-time constant.)

5 tools confirmed in `tools/list` response: `architect_guard`, `block_env_edit`, `block_unsafe_merge`, `session_banner`, `why_blocked`.

## Việc 3 — version bump + exclude

**Version:** `0.1.0` → `0.8.0`. CHANGELOG head was v0.7.0 (P007), P008 = next bump. `env!("CARGO_PKG_VERSION")` in `get_info()` now resolves to "0.8.0" at compile time.

**Exclude list chốt (final Worker decision):**
```toml
exclude = ["scripts/", "docs/", "phieu/", ".sos-state/", ".backup/", ".claude/", ".github/", "hooks/pre-commit", "hooks/pre-push"]
```

**Key discovery — `hooks/` glob collision:** Using `"hooks/"` or `"hooks"` as exclude pattern also excluded `src/hooks/mod.rs` because Cargo's exclude glob treats the pattern as a subpath prefix match (not anchored to root). The `src/hooks/` directory matched `hooks/` → module file disappeared from package → `cargo build` failed with `E0583: file not found for module hooks`. Fix: enumerate the specific files in root `hooks/` directory (`hooks/pre-commit`, `hooks/pre-push`) rather than using the directory glob.

**Why `tests/` kept:** `cargo publish --dry-run` runs `cargo build` on the packaged crate (not `cargo test`), so `tests/` is not strictly needed for dry-run. However, keeping `tests/` allows downstream users to run `cargo test` after `cargo install --path .` and verify behavior. Phiếu said to iterate dry-run — dry-run passed with `tests/` present.

**Package size result:**
- Before (estimated from phiếu): ~778 KB / ~101 files
- After: 147.9 KB / 25 files (44.6 KB compressed)

`cargo publish --dry-run --allow-dirty` → clean, `warning: aborting upload due to dry run` (expected).
