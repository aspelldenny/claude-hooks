## rmcp API thực dùng

**Macro flag:** `#[tool_router(server_handler)]` — confirmed correct. Flag auto-emits `#[tool_handler]` for tools-only servers. Top-doc in `rmcp-1.7.0/src/handler/server/router/tool.rs` lines 24-33 confirms this exact flag.

**Wrapper types:**
- `Parameters<T>` — at `rmcp::handler::server::wrapper::Parameters` (re-exported via `wrapper.rs` which does `pub use parameters::*`).
- `Json<T>` — re-exported at `rmcp::Json` AND `rmcp::handler::server::wrapper::Json`. Used path: `rmcp::handler::server::wrapper::Json`.

**Import paths used in serve.rs:**
```rust
use rmcp::{tool_router, tool, ServiceExt, transport};
use rmcp::handler::server::wrapper::{Parameters, Json};
use rmcp::handler::server::tool::ToolRouter;
use rmcp::schemars;
```
`rmcp::schemars` must be imported into scope explicitly for `#[derive(schemars::JsonSchema)]` to resolve — the derive macro expands to `schemars::JsonSchema` reference which needs the module in scope. Compiler suggests `use rmcp::schemars;`.

**Method sync vs async:** all 4 tool methods are sync (`fn`, not `async fn`). The `_decide` functions are sync. rmcp macro does not require `async` for sync tools.

**`tool_router` field dead_code:** rmcp macro-generated `ServerHandler` impl accesses the field, but Rust's dead_code lint cannot see through proc-macro expansion. Fixed with `#[allow(dead_code)]` on the struct.

## rmcp `macros` feature

**Finding:** `rmcp = { version = "1.7.0", features = ["server", "transport-io"] }` in original Cargo.toml does NOT include the `macros` feature. The `#[tool_router]` and `#[tool]` proc-macros require `macros` = `["dep:rmcp-macros"]` (Cargo.toml line 86-87 in rmcp crate).

**Action taken:** Added `"macros"` to rmcp features:
```toml
rmcp = { version = "1.7.0", features = ["server", "transport-io", "macros"] }
```

**Pre-authorized:** Phiếu Debate Log pre-auth covers "minimum required features for existing deps". `macros` is a feature of the already-present `rmcp` dep, not a new crate. No new crate added.

**No escalation needed** for tokio features — Cargo.toml existing features are sufficient.

## Decision-core refactor impact

**All 81 pre-P006 tests pass unchanged.** This was the primary verify-cò for the refactor.

**Test strategy:** all 81 existing tests go through CLI binary (via `assert_cmd`), not internal functions. The wrapper functions (`architect_guard() -> i32`, etc.) preserve their stdin-read + eprintln + exit-code semantics exactly. Refactor is purely mechanical.

**`io::block` removed:** `pub fn block(reason: &str) -> i32` in `src/io.rs` was unused after refactor (old code called `return io::block(&msg)` from wrappers, but new `_decide` functions build `Decision` directly, and new wrappers use `eprintln!(reason)` inline). Removed to pass `cargo clippy -- -D warnings`. Not a public API breakage (binary crate, no downstream).

**Override marker behavior note:** `block_unsafe_merge_decide` returns `Decision { exit_code: ALLOW, blocked: false, reason: Some(warning_msg) }` for override markers. CLI wrapper prints ANY non-None reason via `eprintln!`. This matches original behavior (override → print warning → allow), verified by `p004_override_marker_allows` test passing.

**`render_banner()` output format:** changed from `println!` series to building a `String` with `push_str`/`push`. Output is identical — verified by `p005_banner_with_backlog_shows_markers` and related tests all passing.

## Handshake smoke approach

**Framing confirmed:** rmcp stdio transport uses **newline-delimited JSON** (one JSON object per line). Confirmed via `rmcp-1.7.0/src/transport/async_rw.rs` (LinesCodec-style newline scanning).

**Test approach:** `std::process::Command::new(binary).arg("serve").stdin(Stdio::piped()).stdout(Stdio::piped())`. Write 3 lines: `initialize` request → `notifications/initialized` notification → `tools/list` request. Drop stdin (EOF) → server `waiting()` returns → process exits → assert stdout contains 4 tool names + `"jsonrpc"` key.

**assert_cmd not used** for the handshake test (assert_cmd write_stdin is one-shot; we needed multi-line sequential write + explicit stdin drop). Used `std::process::Command` directly with `Stdio::piped()` and manual `writeln!` + `drop(child.stdin.take())`.

**No seam needed:** the server exits cleanly on stdin EOF. `waiting()` returns on transport close. 10-second timeout guard prevents test hang if server hangs.
