## Fail-CLOSED Divergence (INTENTIONAL — document for future hooks)

`block-unsafe-merge` is the **only hook in this binary that fails CLOSED** (exit 2 when `gh pr diff` fails or returns empty). The other three real hooks (`architect-guard`, `block-env-edit`, and `session-banner` when ported) all fail **open** (exit 0 on parse/IO error).

**Rationale (oracle L69-82):** When the hook cannot verify whether a PR touches security surface, allowing the merge silently would defeat the purpose of the guard. An unverifiable merge must be blocked so a human can verify manually. This is an explicit security design choice, not an accident.

**How it manifests in code:**
- `gh pr diff` fail or empty stdout → `BLOCK` (exit 2) with verbatim message telling user to check `gh auth status`.
- `gh pr view` fail → empty string → `verdict_is_approve` returns `NoBlock` → falls through to the "no review yet" BLOCK path. This is also correct: a PR that touches security surface but has no review comment must be blocked.

**Maintenance warning:** Do NOT "fix" this to fail-open. Any PR touching this logic requires a phiếu with explicit acknowledgment of the fail-closed doctrine.

## gh Test Strategy

Two-tier strategy per phiếu constraint (no mocking dep):

**Tier 1 — Pure function unit tests (22 cases, no gh required):**
All 4 pure helpers (`parse_merge_pr`, `extract_skip_marker`, `touches_security_surface`, `verdict_is_approve`) are isolated in `#[cfg(test)] mod tests` inside `src/hooks/mod.rs`. These cover all logic branches deterministically.

**Tier 2 — Integration tests (4 gh-free cases, `tests/cli.rs`):**
Covers oracle L31 (empty stdin), L42/L47 (non-merge command, branch-only bypass), L54 (override marker). These paths return ALLOW before any gh call so they work in CI without auth.

**gh-dependent paths (fail-CLOSED BLOCK, verdict):** manual verification only. No seam added; no mocking dep added. Decision: the pure-function decomposition provides adequate automated coverage for the logic; the gh-shelling surface is thin (2 `Command::new("gh")` calls, straightforward) and best verified manually against the Bash oracle.

No mock seam was created. If a future phiếu requires automated testing of gh-failure paths (e.g., to catch regression after refactor), it would need either a `gh` stub binary on `PATH` in CI, or an injected closure seam — both are Tầng 1 decisions requiring a new phiếu.

## env-fallback `CLAUDE_TOOL_INPUT` Not Ported

Oracle L24-28 includes a fallback: if stdin is not a terminal, read stdin; else fall back to `$CLAUDE_TOOL_INPUT` env var. This env-fallback is **not ported** — same decision as P002 and P003. Claude Code always pipes stdin to PreToolUse hooks; the env fallback is an edge-case path that would require a shared `io.rs` harness change (Tầng 1, out of scope for port phiếu). Logged in CHANGELOG.md under "conscious parity gap."

## `grep -A 50` Verdict Limit Edge

Oracle L99 uses `grep -A 50` (50 lines after `<!-- security-review-start -->` marker). Port uses `.take(50)` on the lines iterator after the marker. Edge: if a PR has two consecutive `/security-review` blocks in comments, the first block's verdict governs (both oracle and port). This is correct behavior — the first review block takes precedence. The 50-line limit is unlikely to cause false negatives in practice (security review comments are structured and concise), and the port matches oracle semantics exactly.
